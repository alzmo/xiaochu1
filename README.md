# xiaochu1

### 项目说明
本项目是一个基于 **微信小游戏（Game）** 的三消式分层点击玩法原型，核心目标是围绕“可解性”构建关卡和局面分析能力，而不是仅做静态随机局。当前技术架构为：`game.js` 入口 + Canvas 2D 渲染 + 纯 JS 规则/求解模块，不依赖小程序页面（WXML/WXSS）体系。

### 核心玩法
- **分层方块**：方块带有 layer，只有未被覆盖的顶层方块可点击。  
- **暂存槽**：点击方块先进入暂存槽，槽位有上限。  
- **收纳框**：同色方块会自动进入对应收纳框，达到容量后清空并刷新。  
- **胜负规则**：
  - 胜利：画布清空 + 暂存槽为空 + 所有收纳框 `count = 0`。
  - 失败：暂存槽爆满且无法收纳、无可点击方块，或“提示一步”确认无解且无法通过帮助回正。

### 关键系统设计（必须突出）
- **可解关卡生成（solver）**：
  - 关卡生成后会进入 solver 校验，只有确认可解才作为正式关卡投入游玩。
  - solver 已升级为三态：`solvable / unsolvable / unknown`，其中 `unknown` 代表搜索预算不足，不等价于无解。
- **生成 → 验证闭环流程**：
  1. 生成关卡布局与颜色分布（满足 3 的倍数约束）。
  2. 运行 solver 验证至少存在一条可行路径。
  3. 通过后进入当前关卡，同时预生成下一关。
  4. 局内继续使用同一 solver 做“局面分析/提示/恢复建议”，并将“建议判断”与“帮助执行”解耦，保证规则一致性。

### 项目当前状态
#### 已完成
- 小程序结构已迁移为微信小游戏结构（`game.js` + `game.json` + Canvas 渲染）。
- 核心玩法已跑通：分层点击、暂存槽、收纳框自动处理、胜负判定。
- 三态 solver 已完成：`solvable / unsolvable / unknown`。
- 三态提示系统已完成：
  - 可解：给出下一步推荐并高亮方块；
  - 无解：先评估“加收纳篮 / 加暂存槽”能否回正，再给出恢复建议或明确无解；
  - 无法判断：提示“当前局面较复杂，暂时无法判断是否还有解”。
- 提示一步恢复建议分析链路已修复：
  - `requestHintStep` 在 `UNSOLVABLE` 分支会重新执行“加收纳篮 / 加暂存槽”回正评估；
  - 仅当两种帮助都不能回正时才会触发无解标记；
  - `addExtraBinHelp / addExtraTempSlotHelp` 保持纯执行，不承担 solver 分析职责。
- 帮助系统回归修复已完成：
  - “加收纳篮”会在当前局真实新增 1 个收纳篮，仅本局生效；
  - “加暂存槽”会在当前局真实新增 1 个暂存槽位，仅本局生效；
  - 过关或重开后恢复默认资源上限。

### 运行方式
1. 使用微信开发者工具导入本仓库。
2. 项目类型选择 **小游戏（Game）**。
3. 入口文件为 `game.js`，配置为 `game.json`。
4. 可使用测试 appid（如 `touristappid`）进行本地调试。

### 项目结构
- `game.js`：小游戏启动入口、主循环与触摸事件分发。
- `game.json`：小游戏基础配置。
- `src/levelGenerator.js`：关卡布局与颜色数据生成。
- `src/levelPreparation.js`：当前关卡/下一关预生成与切换调度。
- `src/gameRules.js`：可见性、自动收纳、胜负判断、快照克隆等底层规则。
- `src/solver.js`：三态求解器（可解 / 无解 / 无法判断）与路径返回。
- `src/gameState.js`：局面分析、提示系统、恢复建议时机控制、帮助执行与关卡状态管理。
- `src/renderer.js`：Canvas 2D 渲染、推荐方块高亮、帮助按钮与状态文案绘制。
- `src/storage.js`：关卡进度本地存档。
